# Production Automation Architecture

Date: 2026-07-09

## Purpose

This document defines how DTC becomes a production-ready automated product without turning into an unsafe autopilot bot.

The goal is:

```text
Automate capture, detection, prioritization, drafting, briefing and logging.
Keep judgement, approval, disputes and sensitive customer communication under owner control.
```

## Product principle

DTC is a Business Execution OS.

Automation must strengthen:

```text
Assessment → Diagnosis → Today action → Daily rhythm
```

If automation does not strengthen that chain, it waits.

## Final automation flow

```text
Sources
→ source_connections
→ sync_runs
→ normalized records
→ action engine
→ Today actions
→ notification_queue
→ owner approval / manual action
→ action_audit_log
→ analytics + feedback
```

## Tables introduced in Cycle 13

### `automation_settings`

One row per business.

Controls:

- daily brief enabled/disabled
- daily brief time
- timezone
- preferred brief channel
- customer message mode
- approved-send flag
- autopilot flag
- quiet hours
- max customer messages per day
- pause automation until date

Important default:

```text
customer_message_mode = draft_only
require_approval_for_customer_messages = true
approved_send_enabled = false
autopilot_enabled = false
```

### `source_connections`

Tracks connected sources:

```text
manual
csv
google_sheets
gmail
solobid
rentease
radflow
other
```

This table stores only connection metadata and safe config. Secrets/tokens must not be stored here in plain text.

### `sync_runs`

Every import/sync gets a run record.

Tracks:

- source
- run type
- status
- records seen
- records created
- records updated
- actions created
- error message
- metadata

This makes automation debuggable.

### `notification_queue`

Holds messages that DTC wants to send or show.

Channels:

```text
email
whatsapp
in_app
webhook
```

Default customer-facing behavior:

```text
requires_approval = true
```

This table is not permission to auto-send. It is the queue where future approved-send automation will operate.

### `action_audit_log`

Records important action changes:

- action completed
- action snoozed
- action dismissed
- generated by engine
- notification queued
- notification approved
- notification sent
- automation blocked

This makes DTC trustworthy when automation starts doing more work.

## Automation stages

### Stage 1 — Capture automation

Already started:

- manual quick-add
- CSV / paste import

Future:

- Google Sheets connection
- Gmail reading
- specialist app adapters

### Stage 2 — Detection automation

Already started:

- lead response action
- quote follow-up action
- overdue invoice action
- promised payment check
- recurring invoice reminder

Future:

- delivery blockers
- admin/doc blockers
- source-specific events

### Stage 3 — Daily brief automation

Current:

- owner-triggered daily brief preview/test send

Future:

- scheduled brief per business
- `notification_queue` entry created by scheduler
- owner can pause/disable brief

### Stage 4 — Draft automation

Current:

- WhatsApp draft link
- copy draft button

Future:

- template library
- tone controls
- per-action draft history
- safer email draft flow

### Stage 5 — Approved-send automation

Future only.

Flow:

```text
DTC queues drafts
→ owner reviews
→ owner approves selected drafts
→ DTC sends
→ DTC logs outcome
```

### Stage 6 — Autopilot

Future only, and limited to safe areas first.

Good autopilot candidates:

- owner daily brief
- internal reminders
- recurring admin reminders
- non-sensitive status summaries

Blocked autopilot candidates:

- customer payment chasing
- debt collection wording
- legal threats
- WhatsApp spam
- customer messages without approval
- payment/accounting write-back

## Required guardrails

### Customer-facing messages

Customer-facing WhatsApp/email must remain blocked unless:

```text
requires_approval = true
or approved_at is not null
```

### Quiet hours

No customer-facing send should occur in quiet hours unless the owner explicitly overrides it.

### Pause switch

If `pause_until` is in the future, scheduled automation should skip the business.

### Disputes

Future disputed invoices or sensitive cases must block approved-send automation.

### Opt-out

Future opt-out support must block customer sends.

### Rate limits

Use `max_customer_messages_per_day` before approved-send automation sends anything.

## Production scheduler design

The safest future scheduler sequence:

```text
1. Cron job starts.
2. Load businesses with automation_settings.
3. Skip paused businesses.
4. Run engine.
5. Create owner brief queue item.
6. Never send customer messages without approval.
7. Log sync_runs and action_audit_log entries.
8. Report errors to admin dashboard.
```

## What this cycle does not do

Cycle 13 does not:

- schedule cron jobs
- send customer messages
- connect Gmail/Sheets
- store OAuth tokens
- auto-approve drafts
- bypass RLS
- introduce service-role use in the app runtime

It only creates the production automation backbone.
